# Board Animation And Audio Design

## Goal

为当前中国象棋前端增加接近传统象棋软件的走子动画和音效反馈，同时保持现有房间、走子、托管 `picoclaw`、观战与主持逻辑不变。

第一版目标：

- 棋子不再整盘瞬移刷新，而是按最近一步 `last_move` 播放移动动画
- 普通走子、吃子、终局三类事件提供音效反馈
- 人类手动走子与托管选手走子走同一套前端展示管线
- 维持现有轮询同步架构，不引入 WebSocket 前端实时推送

第一版不做：

- 将军特效
- 粒子特效
- 复杂回放模式
- 自定义音量面板
- 乐谱级拟真音效资源系统

## Problem

当前前端每次拿到新的 `publicMatch.board_rows` 后会直接重绘整张棋盘：

- 棋子在视觉上是瞬时跳变
- 吃子没有专门反馈
- 对局节奏缺少象棋软件常见的“提子、落子、收子”观感
- 即使解说台已经改成赛事播报，棋盘本体仍然偏静态面板

这个问题属于前端展示层，不属于比赛规则层或 agent 接入层。

## Scope

In scope:

- 棋子图层重构
- 基于 `last_move` 的单步动画
- 吃子淡出或缩放消失效果
- `Web Audio API` 合成音效
- 动画期间的轮询状态协调
- 桌面与移动端自适应坐标换算

Out of scope:

- 服务端接口变更
- 比赛逻辑变更
- agent prompt 变更
- 对局规则变更

## Design Summary

前端应将当前“格子内直接渲染棋子”的方式，升级为：

- 棋盘格层：负责网格、河界、选中和落点高亮
- 棋子图层：负责棋子实体的绝对定位与移动动画
- 音效层：负责对局交互音效的统一触发

核心原则：

1. 服务端 `publicMatch` 仍然是唯一真实状态
2. 前端维护一份“稳定显示的基线棋盘”
3. 新走子到达时，前端从“旧基线”过渡到“新真实状态”
4. 动画结束后再提交新的稳定基线

这允许前端在不改后端的前提下，获得平滑视觉效果。

## State Model

建议在前端状态中新增三层棋盘相关状态。

### 1. Server Snapshot

即当前已经存在的：

- `publicMatch.board_rows`
- `publicMatch.last_move`
- `publicMatch.status`

这层始终代表服务端真实局面，前端不可修改。

### 2. Display Base State

前端新增一份“当前已稳定显示”的棋盘基线，例如：

- `boardBaseRows`
- `boardRenderPieces`
- `lastAnimatedMove`

这层表示当前动画完成后真正落稳在界面上的棋盘。

### 3. Animation State

前端新增一个动画态对象，例如：

- `boardAnimation`
  - `move`
  - `from`
  - `to`
  - `piece`
  - `capturedPiece`
  - `startedAt`
  - `durationMs`
  - `phase`
- `pendingMatchSnapshot`
- `audioEnabled`

这层只描述瞬时动画过程，不参与比赛逻辑。

## Rendering Architecture

### Board Grid Layer

继续保留当前 `board-grid` 作为棋盘背景和交互层，负责：

- 九列十行格点布局
- 河界和视觉底纹
- 选中起点
- 合法落点
- 最近一步的起点终点高亮

它不再承担棋子实体渲染职责。

### Piece Layer

新增独立棋子层，覆盖在棋盘格层上方：

- 绝对定位
- 每个棋子元素单独存在
- 常态按格点中心摆放
- 动画时只更新移动棋子的像素位置

推荐继续使用 DOM，而不是 canvas。原因：

- 当前代码和交互已经基于 DOM
- 象棋对象量极低，DOM 足够承载
- 不需要重做点击、选中、可达点等交互系统

## Piece Model

每个棋子对象应至少包含：

- `id`
- `piece`
- `row`
- `col`
- `x`
- `y`
- `isMoving`
- `isCaptured`
- `zIndex`

`id` 不应只使用棋子字符，因为同类棋子会重复。建议在每次从基线棋盘生成图层模型时，按：

- 棋子字符
- 初始格点
- 遍历序号

组合生成稳定标识。

第一版无需实现跨多步保真的棋子身份追踪；只要在单步动画期间能唯一识别移动棋子即可。

## Move Detection

动画触发条件只依赖：

- 新的 `publicMatch.last_move`
- 新旧棋盘快照对比

流程：

1. 前端轮询获得新 `publicMatch`
2. 如果 `last_move` 与 `lastAnimatedMove` 相同，则只刷新普通 UI
3. 如果不同，则尝试构造一段单步动画

如果构造失败，应回退为静态刷新，不允许卡住界面。

## Animation Flow

推荐时序如下。

### Phase 1: Pickup

移动棋子轻微放大，阴影增强，表示提子。

建议：

- 时长：`40ms`
- 缩放：`1.06`

### Phase 2: Travel

棋子从起点中心平滑移动到终点中心。

建议：

- 普通走子：`180ms`
- 吃子：`220ms`
- 缓动：`ease-in-out`

### Phase 3: Settle

棋子在终点轻微回弹或缩放归位，形成落子感。

建议：

- 时长：`60ms ~ 80ms`
- 缩放从 `1.03` 回落到 `1`

第一版总时长建议控制在 `240ms ~ 300ms`。

## Capture Handling

吃子不应通过日志猜测，而应由棋盘状态对比得出。

判断方式：

- 从 `last_move` 得到终点格
- 读取动画开始前基线棋盘的终点格
- 如果终点格原本非空，则认为发生吃子

表现建议：

- 被吃棋子在移动棋子到达前，保持原位
- 到达瞬间做短淡出或缩小消失

这样会比直接瞬间删掉更接近传统棋类客户端的观感。

## Positioning

棋子像素坐标不应硬编码。

应基于棋盘容器的实时尺寸动态计算：

- 读取 `board-grid.getBoundingClientRect()`
- 单格宽度 = `rect.width / 9`
- 单格高度 = `rect.height / 10`
- 棋子中心坐标：
  - `x = col * cellWidth + cellWidth / 2`
  - `y = row * cellHeight + cellHeight / 2`

这样可以兼容：

- 浏览器窗口缩放
- 移动端自适应
- 桌面和窄屏布局切换

## Sync And Queueing

轮询式前端最大的风险是新状态覆盖未播完的动画。

因此必须支持一个最小排队机制：

- 如果当前无动画：
  - 立即开始新动画
- 如果当前有动画：
  - 把新 `publicMatch` 存入 `pendingMatchSnapshot`
- 当前动画结束后：
  - 提交新的 `boardBaseRows`
  - 再检查是否有待处理快照

第一版只需要保存“最新的一份待处理快照”，不必做无限队列。

如果短时间内服务端推进多步，前端允许跳过中间某些视觉过程，但必须保证最终盘面正确。

## Human Move Handling

人类提交走子后，前端不应本地乐观落子。

原因：

- 当前服务端才是唯一裁判
- 人类走子也可能因规则被拒绝
- 本地先走再回滚会让动画和错误提示更混乱

正确做法：

- 继续等待服务端返回新的 `publicMatch`
- 由服务端确认后的 `last_move` 触发动画

这样可以保证：

- 人类
- 托管 `picoclaw`
- 主持人观战

看到的是同一套已确认动画。

## Audio Design

第一版建议不引入静态音频资源文件，而用 `Web Audio API` 直接合成。

原因：

- 不增加额外静态资源管理成本
- 不影响当前嵌入式静态资源发布方式
- Linux / Windows 构建流程无需额外打包处理

### Audio Types

至少提供三种音效：

- 普通落子音
- 吃子音
- 对局结束音

### Audio Unlock

浏览器通常要求用户交互后才能播放音频。

因此需要在首次点击、首次键盘交互或首次走子尝试时初始化 `AudioContext`。

如果音频初始化失败：

- 动画仍应继续
- 只静默降级，不应报错打断对局

## Error Handling

以下情况必须有安全降级路径。

### 无法推导动画

如果前后棋盘状态异常，导致无法安全识别移动棋子：

- 直接刷新到最新稳定盘面
- 不播放本步动画

### 容器尺寸不可用

如果棋盘尚未布局完成，无法读取有效尺寸：

- 延后一次渲染
- 或回退为静态重绘

### 音频不可用

如果浏览器不支持、被拦截或初始化失败：

- 不播放音效
- 不影响动画

## Testing

测试应至少覆盖：

1. `last_move` 变化时能生成单步动画
2. 吃子局面能识别 `capturedPiece`
3. 动画进行中新的快照会被暂存
4. 动画结束后会提交新的基线棋盘
5. 音频初始化失败时不阻塞渲染
6. 关闭动画推导时仍能正确静态显示最新盘面

第一版优先做纯函数和状态转换测试，不要求做浏览器像素级截图校验。

## File Plan

建议的文件边界：

- `static/app.js`
  - 比赛状态同步与事件接线
- `static/board-animation.mjs`
  - 棋子模型、动画调度、棋盘状态过渡、坐标换算
- `static/board-audio.mjs`
  - 音效初始化与播放
- `static/style.css`
  - 棋子层、移动态、吃子态样式

## Acceptance Criteria

- 新走子出现时，棋子会播放可见移动动画
- 吃子时有区别于普通走子的视觉与音效反馈
- 终局时有结束提示音
- 动画不会破坏现有轮询同步逻辑
- 人类和托管选手都走同一套动画管线
- 在不支持音频或动画推导失败时，界面仍能正常工作
