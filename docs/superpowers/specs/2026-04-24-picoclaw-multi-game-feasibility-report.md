# Picoclaw 多游戏竞技场可行性报告

## 报告目标

评估基于当前 `claw-chinese-chess-arena` 项目，扩展出面向 `picoclaw` 外部 agent 的多游戏比拼平台是否可行，重点分析以下两个候选方向：

- 斗地主
- 红色警戒

本报告以技术可行性为主，同时补充必要的运营与风险判断。

前提约束：

- 参赛对象默认是通过 `HTTP / WebSocket / message` 接口接入的外部 agent
- 不把本地传统 bot、桌面脚本、外挂式控制程序纳入第一阶段统一架构
- 当前项目作为分析基线，不直接假设重写整个系统

## 结论摘要

结论分级如下：

- 斗地主：高可行，建议作为下一阶段主项目
- 红色警戒：中低可行，适合先做接入预研，不适合作为当前主线

核心原因不是“哪种游戏更热门”，而是当前项目的架构天然偏向：

- 服务端统一裁判
- 离散状态同步
- 回合制推进
- 单步动作校验
- 可审计日志

这组条件与斗地主高度匹配，但与红色警戒这类实时 RTS 的核心运行方式存在明显错位。

## 当前项目可复用能力

当前项目真正可复用的是竞技场外壳，不是象棋规则本身。基于现有实现，可复用能力主要包括：

- 房间、主持人、观战者、席位分配模型
- 比赛生命周期控制：创建、开始、暂停、恢复、重开
- 外部 agent 接入协议
- `message + session + ws` 混合传输与保活
- 主持端诊断、日志、失败回退
- 快照持久化与对局状态恢复
- 基础前端控制台与服务端 API 路由框架

对应代码边界主要集中在：

- [arena.go](/Users/mewcapoo/Project/claw-chinese-chess-arena/arena.go)
- [app.go](/Users/mewcapoo/Project/claw-chinese-chess-arena/app.go)
- [match.go](/Users/mewcapoo/Project/claw-chinese-chess-arena/match.go)
- [pico.go](/Users/mewcapoo/Project/claw-chinese-chess-arena/pico.go)

这些模块已经具备“让外部 agent 参与比赛”的通用壳层能力。

## 当前项目不可直接复用的部分

以下部分与中国象棋强绑定，迁移到新游戏时基本需要重做：

- 游戏规则引擎
- 合法动作生成
- 状态表示方式
- 面向 agent 的 prompt 格式
- 结果裁定逻辑
- 前端棋盘与可视化表现

对应核心集中在：

- [engine.go](/Users/mewcapoo/Project/claw-chinese-chess-arena/engine.go)

因此，后续如果要做多游戏平台，正确方向不是“继续堆新规则进象棋工程”，而是把当前项目拆成两层：

- 通用竞技场内核
- 各游戏自己的规则适配层

## 三种扩展路线

### 方案 A：服务端裁判型竞技场

平台自己维护完整游戏状态，所有动作都由服务端校验并裁定。

适合：

- 斗地主
- 卡牌博弈
- 棋类
- 桌游类回合制项目

优点：

- 公平性最好控制
- 日志与回放天然可做
- agent 输出格式容易收敛
- 平台可以直接判定非法动作、超时、掉线

缺点：

- 需要自己实现完整规则引擎
- 多人游戏的隐藏信息处理更复杂

结论：

- 斗地主应优先采用该方案

### 方案 B：外部游戏适配层竞技场

平台不负责完整游戏规则，只负责赛事编排、agent 接入、结果采集与诊断。游戏本体由外部引擎、专用服务器或 mod 层负责运行。

适合：

- 红色警戒
- 已有成熟游戏本体的竞技系统

优点：

- 不需要在平台里重写整套游戏
- 能保留原游戏生态

缺点：

- 强依赖目标游戏是否提供稳定的控制接口、观战接口、结果接口、回放接口
- 环境一致性、同步、版本控制复杂
- 裁判权不完全掌握在平台侧

结论：

- 红色警戒如要推进，应先采用该方案做预研

### 方案 C：桌面自动化型竞技场

通过读屏、OCR、模拟鼠标键盘的方式驱动 agent 参赛。

优点：

- 理论上适配面最广

缺点：

- 稳定性差
- 难以调试
- 延迟高
- 反作弊弱
- 几乎无法做强裁判

结论：

- 不建议作为正式项目主线

## 斗地主可行性分析

### 技术匹配度

斗地主与当前竞技场架构的匹配度高，原因如下：

- 天然回合制
- 状态离散且可完整序列化
- 动作集合有限，易做服务端校验
- 规则明确，裁判权可放在平台
- 比赛日志与回放容易实现

这意味着当前项目的“房间、主持、接入、日志、保活、观战”框架可以保留，大部分新增工作集中在游戏层。

### 需要新增的核心能力

- 三人席位模型，不再是红黑双边
- 发牌与洗牌机制
- 叫分、抢地主、加倍、明牌等阶段机
- 每位选手的私有手牌视图
- 公共出牌区与回合上下文
- 牌型识别与合法动作枚举
- 胜负、春天、反春、倍数与积分结算

### 主要技术难点

- 隐藏信息隔离
- 三人轮转而非双人对抗
- prompt 长度和决策上下文控制
- 牌型判断和合法动作生成容易出错
- agent 串通与异常行为审计

### 风险判断

这些难点都是真问题，但都属于规则引擎和平台治理问题，不是架构性死结。只要服务端是唯一裁判，工程风险可控。

### MVP 判断

斗地主适合做 MVP，建议第一版严格收敛范围：

- 仅支持经典斗地主三人局
- 不做复杂赛事系统，先做单局和房间赛
- 不做开放插件协议，先固定 `picoclaw` 接入
- 不做复杂经济系统或排名系统

## 红色警戒可行性分析

### 技术匹配度

红色警戒与当前项目的匹配度低于斗地主，原因如下：

- 它是实时系统，不是回合制
- 动作空间是连续高频的，不是单步动作
- 局势观测不是单个快照能表达完整
- 比赛裁判通常依赖外部游戏引擎而非平台自身

这意味着当前项目的核心可复用部分只剩：

- 比赛编排
- agent 接入管理
- 会话保活
- 日志与诊断面板

而最难的部分都不在当前工程里。

### 红警真正的决定因素

红警是否可做，不取决于你愿不愿意投入，而取决于是否具备稳定接入层：

- 是否有可控的独立运行环境
- 是否能拉起对局实例
- 是否能订阅局势数据
- 是否能向游戏发送动作命令
- 是否能稳定拿到比赛结果和回放
- 是否能保证不同参赛 agent 运行环境一致

如果这些条件不满足，项目会退化成“桌面自动化+读屏控制”，那会直接落入方案 C 的问题集合。

### 主要技术难点

- 实时状态同步和时延控制
- 大量连续动作的指令协议设计
- 游戏实例编排、资源占用和故障恢复
- 判负、超时、异常退出、断线重连
- 回放、审计、反作弊
- 版本统一和 mod 兼容性

### 风险判断

红警的最大风险不是“做不出界面”，而是：

- 你可能做出一个能跑 demo 的系统
- 但做不出可稳定运营的竞技场

也就是说，概念验证可能不难，但产品化门槛很高。

### MVP 判断

如果坚持做红警，第一阶段也不应该叫“正式竞技场”，而应该叫：

- 红警 agent 接入预研
- 红警比赛编排原型

MVP 目标只能是验证三件事：

- 能否稳定拉起比赛
- 能否稳定读局势并发指令
- 能否稳定回收结果

在这三项没有成立前，不建议定义更大范围。

## 技术维度对比

| 维度 | 斗地主 | 红色警戒 |
| --- | --- | --- |
| 与现有架构匹配度 | 高 | 低 |
| 服务端裁判可行性 | 高 | 低 |
| 动作空间复杂度 | 中 | 极高 |
| 状态表示复杂度 | 中 | 极高 |
| 隐藏信息处理 | 高 | 低到中 |
| 实时性要求 | 低到中 | 极高 |
| 接入层不确定性 | 低 | 极高 |
| 运营稳定性预期 | 高 | 低到中 |
| 适合作为下阶段主项目 | 是 | 否 |

## 运营与风险补充

### 斗地主

运营侧风险主要包括：

- agent 串通
- 洗牌公平性争议
- 非法信息泄露争议
- 对局复盘与申诉需求

但这些风险可以通过：

- 服务端洗牌
- 严格私有视图隔离
- 完整动作日志
- 可回放对局记录

进行控制。

### 红色警戒

运营侧风险主要包括：

- 比赛环境不一致
- 地图、版本、mod 差异
- 实例异常退出
- 资源占用过高
- 结果不可复核
- 接口不稳定导致比赛中断

这类问题一旦出现，通常不是补一个接口就能解决，而是需要整套运行平台工程。

## 推荐路线

推荐按以下顺序推进：

1. 把当前项目抽象为“通用竞技场内核 + 象棋游戏适配层”
2. 在该内核之上立项斗地主竞技场
3. 等斗地主版本稳定后，再单独做红警接入预研

不建议的路线：

- 同时并行做斗地主和红警
- 直接在当前象棋工程里硬塞第二种游戏
- 在没有稳定红警接入层前就定义正式竞技产品

## 实施建议

如果要进入下一步，建议把后续工作拆成两个阶段。

### 阶段一：平台抽象

目标：

- 把当前象棋专用工程拆出通用竞技场壳层
- 明确游戏适配接口

产出应包括：

- 通用 `Room / Match / Seat / AgentRuntime` 模型
- 游戏适配接口定义
- 通用日志、快照、保活接口

### 阶段二：斗地主 MVP

目标：

- 基于统一竞技场内核落地首个三人制游戏

产出应包括：

- 斗地主规则引擎
- 三席位视图与主持页面
- picoclaw 斗地主协议
- 基础观战与回放

红警不应并入这个主计划，而应单独立“接入预研”分支。

## 最终建议

最终建议如下：

- 立项：可以
- 主方向：斗地主
- 红警策略：先预研，不正式立项为主线
- 当前工程改造重点：先做平台通用化，再做第二个游戏

一句话结论：

当前项目适合演化成 `picoclaw` 的多游戏竞技场内核，但第一个扩展游戏应当是斗地主，不应直接跳到红色警戒。
